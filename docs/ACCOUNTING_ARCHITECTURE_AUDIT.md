# Accounting Architecture Audit – Canonical Source-of-Truth & Safe Repair Plan

**Company context:** `company_id = 'eb71d817-b87e-4195-964b-7b5321b480f5'`  
**Constraints:** No destructive DB changes in this phase. No table drops or deletes. Read-only diagnostics and classification only.

---

## 1. Executive summary

The ERP accounting layer uses **multiple overlapping tables and numbering paths**. The **live** double-entry flow is: **accounts** (Chart of Accounts), **journal_entries** + **journal_entry_lines** (Journal), **payments** (cash/bank movements, Roznamcha source), and **worker_ledger_entries** (Worker Ledger). Studio costs and courier payables are derived from the same journal + accounts. A separate **ledger_master** / **ledger_entries** set exists for supplier/user ledgers but is **not** the source for customer or worker ledgers.

**Findings:**

- **Chart of Accounts:** Live table is **accounts** (company_id, code, name, type). A legacy **chart_accounts** (and **account_transactions**) exists in `supabase-extract` and is **not** used by the current app; it would overlap if both were populated.
- **Payment references (PAY0069 etc.):** Payment refs are generated by **generate_document_number** RPC (table **erp_document_sequences**) for web flows (saleService, purchaseService, UnifiedPaymentDialog). **document_sequences_global** + **get_next_document_number_global** exist for some flows (e.g. CUS, SL). **document_sequences** is still used by refundService, creditNoteService, purchaseReturnService (legacy). Duplicate PAY refs can occur if: (1) the same sequence is used and a ref is reused (e.g. retry without consuming next), (2) worker ledger rows incorrectly use payment_reference (fixed in Complete+Pay Now flow), or (3) multiple sequence sources are used for the same document type.
- **Reports:** Journal/Account Ledger read **journal_entries** + **journal_entry_lines** + **accounts**. Roznamcha reads **payments** only (cash/bank in/out). Worker Ledger reads **worker_ledger_entries**. Studio Costs report uses **journal_entries**, **journal_entry_lines**, **worker_ledger_entries**, and **studio_production_stages**. Customer Ledger uses **sales**, **payments**, **sale_returns**, **rental_payments**, and optionally RPCs **get_customer_ledger_sales** / **get_customer_ledger_payments**. Mismatches (e.g. “Roznamcha does not show some entries from today”) are likely due to: Roznamcha being **payments-only** (no journal-only entries), or date/time filters (payment_date vs entry_date), or branch filter.
- **Courier payables:** Implemented via **accounts** (2030 control, 2031+ per-courier), **journal_entries** (reference_type = 'shipment' / 'courier_payment'), and views **courier_ledger**, **courier_summary**, **shipment_ledger** (from journal_entry_lines + journal_entries + accounts).

**Recommended direction:** Canonicalize on **accounts**, **journal_entries**, **journal_entry_lines**, **payments**, **worker_ledger_entries**, and **erp_document_sequences** for numbering. Treat **document_sequences** and **document_sequences_global** as legacy; migrate remaining callers to **generate_document_number** (erp_document_sequences) where appropriate. Do **not** drop or truncate any table in this phase; only add comments and a cleanup roadmap.

## 4. Chart of Accounts integrity findings

- **Duplicate account codes:** Prevented by **UNIQUE(company_id, code)** on **accounts** in 03_frontend_driven_schema. Diagnostic 1 in `chart_of_accounts_audit.sql` should return 0 rows.
- **Duplicate names (different ids):** Possible; diagnostic 2 lists them (informational).
- **Parent/child:** **accounts.parent_id** references **accounts(id)**. Diagnostic 3 finds children whose parent is missing or in another company.
- **Control accounts:** 2030 (Courier Payable) is control; 2031+ are per-courier. Worker Payable 2010 is a control; posting is to journal and worker_ledger_entries. Diagnostic 6 lists 203x and 2010 and contact_id usage.
- **Missing defaults:** Diagnostic 4 checks for 1000, 1010, 2000, 2010, 2030, 5000, 4100, 5100. Ensure accounting_ensure_default_accounts / accounting_ensure_erp_accounts_rpc have been run for the company.
- **Orphan accounts:** Diagnostic 5 lists accounts never used in journal_entry_lines (may be intentional for new or control-only).
- **Courier/worker/studio overlap:** 203x and 2010 and contact_id accounts; no structural overlap if 2010 is single and 203x are per-courier.

## 5. Payment numbering findings

- **Canonical sequence:** **erp_document_sequences** with **generate_document_number(company_id, branch_id, document_type, p_include_year)**. Used by documentNumberService.getNextDocumentNumber → saleService, purchaseService, UnifiedPaymentDialog (payment ref), studioProductionService (job), settingsService.getNextDocumentNumber (payment, sale, purchase, etc.).
- **Multiple generators:**  
  - **document_sequences_global** + **get_next_document_number_global**: used for CUS (contactService), SL (quotationService), and fallback in studio*InvoiceService.  
  - **document_sequences**: used by refundService, creditNoteService, purchaseReturnService (and possibly product SKU).  
  So PAY can in theory come from erp_document_sequences (web payment) or, if any path used document_sequences_global with type PAY, from there. Duplicate PAY references can occur if: (1) frontend calls getNextDocumentNumber but then retries and reuses the same ref without incrementing again; (2) DB trigger sets reference_number and app also sets it (double write); (3) worker_ledger_entries incorrectly stored PAY ref on job rows (addressed in Complete+Pay Now fix).
- **Same PAY0069 multiple times:** Run `payment_reference_sequence_audit.sql` (section 1): duplicate reference_number in **payments**; and section 6: same payment_reference in **worker_ledger_entries**. If duplicates exist, cause is either duplicate payment insert (retry/race) or worker ledger using same ref for multiple rows (logic fixed in app; existing data may need repair scripts already in scripts/worker_ledger_repair).
- **Recommendation:** Use **generate_document_number** (erp_document_sequences) as the only source for PAY. Ensure payment insert uses a single code path (e.g. documentNumberService.getNextDocumentNumber('payment') then insert with that reference_number). Do not call get_next_document_number_global for PAY. Migrate refund/credit/return numbering to erp_document_sequences when feasible.

## 6. Report mismatch findings

- **Worker Ledger “behaves better”:** After fixes: (1) idempotent recordAccountingPaymentToLedger (skip if row exists for same journal), (2) display dedupe by (reference_type, reference_id) in getWorkerLedgerData / getWorkerLedgerEntries, (3) Complete+Pay Now: job row vs payment row separated and PAY ref not used for job row. Remaining risk: old duplicate rows in DB (scripts/worker_ledger_repair for cleanup).
- **Studio Costs “misses payment side”:** Studio cost is posted as Dr 5000 Cr 2010 (and worker_ledger job row). Payment is Dr 2010 Cr Cash + worker_ledger payment row (or markStageLedgerPaid). If the report only sums journal lines for “studio” reference_type, it may not include the payment (Dr 2010 Cr Cash) in the same “studio” bucket; payment is reference_type worker_payment. So “payment side” may appear under a different filter (e.g. by account 2010 or by worker_payment). Recommendation: define “Studio Costs” report to include both (1) studio_production_stage journal lines and (2) worker_payment lines that reference the same company/branch/period, or add a “Studio payments” section that reads worker_ledger_entries (reference_type = accounting_payment) for workers used in studio.
- **Journal/Journal Entries partial or inconsistent:** Possible causes: (1) branch filter (user sees only one branch); (2) date filter; (3) filtering out deleted purchases/sales in accountingService.getAllEntries; (4) balance trigger (accounting_validate_journal_balance_trigger) rejecting or not running. Run report_source_reconciliation.sql (sections 1–2) to check entry counts and balance per entry.
- **Roznamcha “does not show some entries from today”:** Roznamcha is **payments-only**. Journal entries that do not have a corresponding row in **payments** (e.g. manual journal, accruals, or entries created by RPC without a payment record) will not appear. So “entries from today” from the Journal will only appear in Roznamcha if they are cash/bank movements recorded in **payments**. Recommendation: either (1) document that Roznamcha is cash/bank movements (payments) only, or (2) extend Roznamcha to also show journal lines for Cash/Bank accounts (with a clear source label) and reconcile with payments.

## 8. Safe cleanup roadmap (non-destructive)

- **Do not delete or drop tables in this phase.**
- **Legacy tables to mark (comment only):**  
  - **document_sequences:** Already commented in erp_legacy_table_comments.sql (“LEGACY: Prefer erp_document_sequences…”).  
  - **document_sequences_global:** Add COMMENT: “Legacy/alternate numbering for CUS, SL, etc. Prefer erp_document_sequences + generate_document_number for PAY, SL where possible.”  
  - **chart_accounts**, **account_transactions:** COMMENT: “Legacy schema (supabase-extract). Live Chart of Accounts is accounts. Do not use for posting.”
- **Duplicate tables to review later:** **worker_payments** (if it exists): verify if any code still inserts/reads it; if not, mark legacy and plan archive-after-verification.
- **Archive-first candidates (after verification and backup):** None in this phase. Future: document_sequences rows could be migrated to erp_document_sequences and then table deprecated (not dropped until all callers migrated).
- **Tables that must NOT be touched:** **accounts**, **journal_entries**, **journal_entry_lines**, **payments**, **worker_ledger_entries**, **erp_document_sequences** (except normal upserts from app/RPC).
- **Prerequisites before any cleanup:** (1) All payment numbering call paths use erp_document_sequences for PAY. (2) Customer ledger and report screens verified to use intended source (sales + payments + RPC or direct). (3) Worker ledger repair scripts run and verified for company. (4) Full DB backup and restore test. (5) No open retries or duplicate inserts (idempotent patterns in place).

## 9. High-risk areas not to touch yet

- **accounts:** Do not merge or rename accounts used in journal_entry_lines without a full migration plan (repoint lines, then update accounts).
- **journal_entries / journal_entry_lines:** Do not delete or bulk-update; balance trigger and reporting depend on them. Void entries via status/flag if supported, not delete.
- **payments:** Do not change reference_number or delete payments that are linked to journal_entries (payment_id). Dedupe only with explicit business approval and script that also fixes journal references.
- **worker_ledger_entries:** Run only read-only and approved repair scripts (e.g. scripts/worker_ledger_repair); do not ad-hoc delete or change reference_type/reference_id.
- **erp_document_sequences / document_sequences_global / document_sequences:** Do not reset or truncate sequences without a formal numbering reset process (risk of duplicate document numbers).
- **ledger_master / ledger_entries:** If used for supplier/user ledger UI, do not drop; clarify with product whether this is still required before deprecating.

## 10. Recommended next repair phase

1. **Run diagnostics:** Execute all four SQL files in docs/audit/ for the company. Record: duplicate account codes (expect 0), duplicate payment reference_number, sequence last_number vs max PAY in payments, journal balance imbalances, and row counts.
2. **Canonicalize payment numbering:** Ensure every payment insert path (sale, purchase, worker Pay Now, expense, etc.) uses **documentNumberService.getNextDocumentNumber(companyId, branchId, 'payment')** (i.e. **generate_document_number** → erp_document_sequences). Remove or avoid using get_next_document_number_global for PAY. Add a single place that assigns payment reference_number on insert (no DB trigger overwriting with a different sequence).
3. **Add table comments:** Apply COMMENT on document_sequences_global, chart_accounts, account_transactions, and worker_payments (if exists) as in section 8.
4. **Studio Costs report:** Define and implement “payment side” (worker payments against studio jobs) in the report (journal worker_payment lines and/or worker_ledger_entries accounting_payment for studio workers).
5. **Roznamcha:** Either document as “payments only” or extend to include Cash/Bank journal lines with a clear source label.
6. **Customer ledger:** Confirm whether frontend uses get_customer_ledger_sales / get_customer_ledger_payments RPC or direct sales/payments queries; document and make consistent.
7. **Worker ledger:** For the audited company, run duplicate detection/cleanup from scripts/worker_ledger_repair if diagnostics show duplicate (reference_type, reference_id) or duplicate payment_reference; then re-run report_source_reconciliation and payment_reference_sequence_audit to confirm.

After the above, plan a second phase: migrate document_sequences callers (refund, credit note, purchase return) to erp_document_sequences, then deprecate document_sequences and document_sequences_global for PAY/SL/PUR (archive-only, no drop).
